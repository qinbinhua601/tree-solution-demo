# Headless Tree 分页实现分析

## 结论

当前实现的思路可以复用到其他 `headless tree` 项目，但这份代码本身不是可直接复制的通用模块。

更准确地说：

- 可复用的是“根节点本地渐进分页”的方案
- 不可直接复用的是当前项目里写死在业务 tree hook 里的实现细节

## 当前实现是什么

当前项目不是服务端真分页，也不是把分页能力接进 `@headless-tree` 内部。

它做的是：

1. 根节点首次加载时一次拿到完整一级节点
2. 用 `visibleRootCount` 控制当前只显示前 N 个根节点
3. 组件渲染时对根节点集合做 `slice`
4. 底部哨兵进入视口时，扩大可见窗口

所以它本质上是：

- 数据层全量加载
- 展示层渐进展开
- 触发层自动触底加载

## 哪些部分可复用

- 只对根节点做分页展示，子节点继续使用原有懒加载
- 用 `visibleCount` 管理完整集合和当前显示窗口
- 用底部哨兵 + `IntersectionObserver` 触发“加载更多”
- 在根节点新增、删除、拖拽进出根目录时同步分页状态

这类方案适合：

- 根节点很多，但子节点仍按需展开
- 目标是减少首屏渲染量，而不是实现服务端真分页
- 不想改动 `headless tree` 内部数据模型

## 哪些部分不通用

- 只处理根节点，没有抽象成任意层级分页
- 分页状态直接写在当前业务 tree hook 中
- 页大小、加载节奏和状态同步方式都带有明显项目耦合
- CRUD 和拖拽都需要手动维护分页状态

所以当前代码更适合作为“参考实现”，不适合作为“直接迁移的基础模块”。

## 给其他 headless tree 项目复用时的建议

如果别的项目也要做同类能力，建议按三层拆：

1. 纯分页层  
   负责完整集合、可见数量、可见数据、`hasMore`、`loadMore`、新增、删除。

2. tree 适配层  
   负责把根节点集合接进分页层，并在根节点 CRUD / 拖拽时同步分页状态。

3. 组件接入层  
   负责滚动容器、底部哨兵、触底加载和底部状态展示。

## 什么时候不要照搬当前方案

以下情况不建议直接复用当前思路：

- 需要服务端真分页，每次滚动都请求下一页
- 需要每一层节点都支持分页
- 希望做成稳定的公共基础能力，而不是单项目方案

这时应该单独设计服务端分页协议和 tree 适配方式，而不是继续沿用“全量加载后再 slice”的做法。
